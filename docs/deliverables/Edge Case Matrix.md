# Edge Case & Mitigation Matrix

| Scenario / Trigger | Expected User Experience | Technical Implementation / Mitigation | Resulting FSM State |
| :--- | :--- | :--- | :--- |
| **Page Refresh during Checkout** <br>*(User hits F5 while on the payment screen)* | The user remains on the checkout page with their secure session intact. | The FSM automatically serializes its status and `idempotencyKey` to `sessionStorage` via a `useEffect`. On mount, the reducer hydrates from this storage, bypassing the cart screen. | Recovers to `CHECKOUT_VALIDATED` |
| **Spamming the "Pay" Button** <br>*(User double-clicks rapidly due to lag)* | Only one transaction is processed; no double-charging occurs. | **UI Layer:** "Pay" button unmounts instantly on click. <br>**FSM Layer:** Reducer ignores subsequent `SUBMIT_ORDER` dispatches if not in `VALIDATED` or `FAILED`. <br>**Network Layer:** `idempotencyKey` prevents duplicate server processing. | `ORDER_SUBMITTED` |
| **Price Tampering / Stale Cart** <br>*(DOM manipulated to change prices, or API prices changed)* | Checkout is halted; user is forced to review the correct, updated prices. | The `VALIDATE_CHECKOUT` action triggers a fresh pull from the IndexedDB/API. If the generated `cartHash` mismatches the expected total, the transaction is actively blocked. | `ORDER_INCONSISTENT` |
| **Cross-Tab Concurrency** <br>*(Tab A proceeds to checkout while Tab B tries to add items)* | Tab B is visually locked and prevented from modifying the cart payload. | Tab A writes a `checkout_lock` to `localStorage`. Tab B's `storage` event listener detects the lock and disables all UI modification buttons. | `LOCKED_BY_OTHER_TAB` |
| **Network Timeout / Gateway Hang** <br>*(Simulated Chaos: API takes >10s to respond)* | The UI presents a safe failure message and offers a retry option without requiring re-validation. | The FSM catches the timeout error. Crucially, the `idempotencyKey` is **not** burned (burned only on success). The user can safely hit "Retry" using the exact same secure token. | `ORDER_FAILED` |
| **Browser Storage Quota Exceeded** <br>*(Incognito mode limits hit during FSM serialization)* | The app safely aborts the checkout rather than resting in a decoupled UI state. | The `sessionStorage` persistence hook is wrapped in a `try...catch`. If a `QuotaExceededError` is thrown, a `PERSISTENCE_FAILURE` is dispatched to safely crash the flow. | `ORDER_INCONSISTENT` |
| **Post-Transaction Modification** <br>*(User tries to click "+" on an item immediately after paying)* | The system forces a clean slate; old tokens cannot be accidentally reused. | The FSM requires an explicit `RESET_CHECKOUT` dispatch (triggered by clicking "Return to Shop") before it accepts `UPDATE_CART` actions, ensuring the token lifecycle is strictly closed. | `CART_READY` |
| **Partial API Response** <br>*(Simulated Chaos: Gateway returns 200 OK but missing receipt ID)* | App rejects the malformed response and prompts for support. | The submission logic validates the structure of the JSON payload. If missing required fields, it throws an `INVALID_PAYLOAD` error and exposes the Diagnostic Copy tool. | `ORDER_FAILED` |
